# LLM 学习问答总结

> 你在学习 LLMs-from-scratch 过程中提出的所有问题及简短回答

---

## 1. 为什么 decode 出来有的字符无法显示（� 符号）？

**原因**：UTF-8 编码是多字节的（1-4字节）。单独解码某个 token 时，如果它是多字节字符的一部分，就会产生 � 符号。

**例子**：
```python
token_233 = "Ġ为"  # UTF-8: [0xE4, 0xB8, 0xBA]
如果只有部分字节 → 无法解码 → �
```

**解决**：完整解码整个 token 序列，而不是单独解码每个 token。

---

## 2. 每一个字符都是 50257 维度吗？

**不对**。输出是 `[batch_size, seq_len, vocab_size=50257]`。

- **每个位置** 输出 50257 个 logits（未归一化的概率）
- **不是**每个字符本身是 50257 维
- 通过 softmax 转换为词表上的概率分布

**类比**：就像多选题，每个位置给出 50257 个选项的分数。

---

## 3. 在梯度演示中，为什么标签是 0？

**这是演示用例**：
- 标签设为 0 只是为了方便计算梯度
- 真实训练中，标签是真实的下一个 token ID
- 损失函数：`loss = (prediction - target)²`
- 当 target=0 时，`∂loss/∂prediction = 2 × prediction`，梯度计算更直观

---

## 4. GPT 训练中的真实 target 是什么？多位置训练如何工作？

**关键洞察**：每个位置都有自己的预测任务。

**例子**：
```
输入 token_ids: [5661, 367, 2885, 1464]
目标 token_ids: [367, 2885, 1464, 1807]

位置 0: 用 5661 预测 367
位置 1: 用 [5661, 367] 预测 2885
位置 2: 用 [5661, 367, 2885] 预测 1464
位置 3: 用 [5661, 367, 2885, 1464] 预测 1807
```

**每个位置都有损失**，总损失是所有位置损失的平均。这叫 **Teacher Forcing**。

---

## 5. 标签里前面三个 ID 不是没用了吗？

**不是**！所有位置都在训练：

```
target_ids = [367, 2885, 1464, 1807]
         ↓       ↓       ↓       ↓
     位置0    位置1    位置2    位置3
     各自的预测目标
```

- 不是只有最后一个 1807 有用
- 每个 ID 都是对应位置的监督信号
- 一个序列训练多个预测任务，效率高

---

## 6. Transformer 的 heads（多头）和卷积核的概念是否类似？

**相似之处**：
- 都是并行学习多种特征
- 多个"专家"各司其职
- 最后融合信息

**关键区别**：
- **卷积核**：固定权重，局部感受野
- **多头注意力**：动态权重（基于输入计算），全局感受野

**类比**：
- 卷积 = 固定模板扫描
- 多头注意力 = 动态查询匹配

---

## 7. tok_emb 和 out_head 的维度关系？

**tok_emb.weight**：`[vocab_size=50257, emb_dim=768]`
- 将 token ID 映射到 768 维向量
- 输入层：查表操作

**out_head.weight**：`[vocab_size=50257, emb_dim=768]`
- 将 768 维向量映射到词表概率
- 输出层：投影到词表

**为什么形状相同但作用不同**？
- tok_emb：ID → 语义空间（768维）
- out_head：语义空间 → 词表分布（50257维）
- 两个是逆操作，所以形状转置关系

---

## 8. LayerNorm 的 scale 和 shift 参数是否参与计算？

**是的**！它们是可学习参数。

**公式**：
```
normalized = (x - mean) / sqrt(var + ε)
output = scale * normalized + shift
```

- **scale（γ）**：学习最优方差
- **shift（β）**：学习最优均值
- 初始化：scale=1, shift=0
- 通过反向传播更新

**参数量**：每个 LayerNorm 有 `2 × emb_dim` 个参数（GPT-2: 1536）

---

## 9. 参数和内存的对应关系？

**基本换算**：
```
内存 (MB) = 参数数量 × 字节数 / 1,048,576
```

**数据类型**：
- float32: 4 bytes
- float16: 2 bytes（节省 50%）
- int8: 1 byte（节省 75%）

**示例（GPT-2 Small, 124M 参数）**：
- float32: 124M × 4 = 496 MB
- float16: 124M × 2 = 248 MB

**训练 vs 推理**：
- **推理**：只需模型参数（~500 MB）
- **训练**：参数 + 梯度 + 优化器状态 + 激活值（~2-4 GB）

---

## 10. 如何修改已训练模型的词表？增加一些词有什么影响？

**步骤**：
1. 扩展 `tok_emb`：复制旧权重，初始化新词 embedding
2. 扩展 `out_head`：复制旧权重，初始化新词输出权重
3. **必须微调**：否则新词效果很差

**代码模式**：
```python
new_tok_emb.weight.data[:old_vocab_size] = old_tok_emb  # 保留旧权重
nn.init.normal_(new_tok_emb.weight.data[old_vocab_size:], mean=0.0, std=0.02)  # 新词
```

**影响**：
- 参数增加：`new_tokens × emb_dim × 2`（tok_emb + out_head）
- 旧词性能可能略有下降
- 需要包含新词的训练数据

---

## 11. qkv_bias 是什么？为什么有这个参数？

**定义**：控制 Q、K、V 线性投影是否使用偏置项。

```python
# qkv_bias = True
y = Wx + b  # 有偏置

# qkv_bias = False
y = Wx      # 无偏置
```

**GPT-2 选择 False 的原因**：
1. **Layer Norm 会中和 bias 的效果**
2. 节省参数（GPT-2 Small: ~27,648 个）
3. 实验表明对性能影响极小
4. 简化模型

**不同模型的选择**：
- BERT: True
- GPT-2/GPT-3: False
- LLaMA: False

## 12. FFN 维度错误：为什么不能用 context_length？

**错误**：
```python
nn.Linear(cfg["context_length"], 4 * cfg["context_length"])  # ❌ 错误
# context_length = 1024，这是序列长度，不是向量维度
```

**正确**：
```python
nn.Linear(cfg["emb_dim"], 4 * cfg["emb_dim"])  # ✅ 正确
# emb_dim = 768，这是每个 token 的向量维度
```

**维度流**：
```
[batch, seq_len, emb_dim]
[2, 3, 768]
    ↓
FFN: 768 → 3072 → 768
    ↓
[2, 3, 768]  # 输出维度必须和输入相同（残差连接要求）
```

**记忆口诀**：
- `context_length`（1024）= 序列有多长
- `emb_dim`（768）= 每个 token 的向量有多少维
- FFN 只处理每个 token 的向量，所以用 `emb_dim`

## 核心概念速查

| 概念 | 要点 |
|------|------|
| **Tokenization** | BPE 算法，UTF-8 编码，token ID ↔ 文本 |
| **Embedding** | token ID → 768维向量（查表） |
| **Position Encoding** | 给每个位置加上位置信息 |
| **Attention** | 动态加权，Q/K/V 投影，多头并行 |
| **FFN** | 768 → 3072 → 768，扩展-压缩模式 |
| **Layer Norm** | 标准化 + 可学习的 scale/shift |
| **输出层** | 768维 → 50257维（词表大小） |
| **训练** | 多位置同时预测，Teacher Forcing |
| **参数量** | 124M 参数 = 496 MB (float32) |
| **维度区分** | emb_dim=768（向量维度），context_length=1024（序列长度） |

3. **可视化**：画出数据流图，标注每一步的维度
4. **小规模实验**：用小数据测试（如 batch=2, seq_len=3）

## 13. 为什么 torch.multinomial 采样只输出 3 种 token？

**原因**：`probas` 概率分布中只有 3 个位置是非零概率。

**数学解释**：
```python
probas = [0.3, 0.5, 0.2, 0.0, 0.0, ...]
         └────有效支撑集────┘  └─概率为0─┘
```

`torch.multinomial` 只能从**非零概率**的位置采样，就像轮盘赌中概率为 0 的扇区永远不会被选中。

**产生原因**：通常是做了 **top-k 或 top-p 过滤**：
- 只保留概率最高的 k 个 token（如 k=3）
- 其余位置概率设为 0
- 重新归一化让保留的概率和为 1

**增加多样性的方法**：
1. 提高温度参数 `temperature > 1`
2. 增大 top-k 值（如从 3 增到 10）
3. 使用 top-p（nucleus）采样

## 14. 温度参数：为什么 logits/0.1 会让概率变得极小？

**困惑点**：`logits / 0.1` 等于 `logits × 10`（放大了），但为什么概率反而变小了？

**关键**：Softmax 的指数函数造成"赢家通吃"效应。

**数学原理**：
$$\text{softmax}(x_i) = \frac{e^{x_i}}{\sum_j e^{x_j}}$$

当输入放大 10 倍时，**指数放大**极其恐怖：
```python
e^60 ≈ 10^26    # 最大值
e^40 ≈ 10^17    # 次大值
e^20 ≈ 10^9     # 第三大

# 分母被最大项主导
softmax ≈ 10^26 / 10^26 ≈ 1.0（最大值）
其他位置 ≈ 10^17 / 10^26 ≈ 10^-9（接近0）
```

**实际效果**：
```python
T=1 (原始):    [0.061, 0.572, 0.358] - 相对平滑
T=0.1 (×10):   [1.8e-10, 0.991, 0.009] - 极度尖锐（低温）
T=5 (÷5):      [0.155, 0.242, 0.220] - 更均匀（高温）
```

**记忆口诀**：
- "温度低了冻一团，温度高了满天飞"
- "低温确定性，高温多样性"

---

## 15. 温度参数的直观记忆方法（顺逻辑）

**最佳记忆法：物理温度类比**

| 温度值 | 物理现象 | 分布形状 | 模型行为 |
|--------|----------|----------|----------|
| **T 大（高温）** | 分子到处乱跑 | **宽** | 不确定，探索 |
| **T 小（低温）** | 分子冻结 | **窄** | 确定，专注 |

**顺逻辑关系**：
$$T \uparrow \Rightarrow \text{分布宽度} \uparrow \quad \text{(同向变化)}$$

**类比**：
- **高温** = 模糊滤镜 = 雨露均沾 = 头脑风暴
- **低温** = 锐化滤镜 = 赢家通吃 = 果断决策

**记忆口诀**：
> "温度跟着宽度走"
> "热胀冷缩"（温度高→分布膨胀，温度低→分布收缩）

## 16. 优化器状态需要保存吗？

**取决于使用场景**：

| 场景 | 是否保存优化器 | 原因 |
|------|---------------|------|
| **中断后继续训练** | ✅ **必须** | 需要恢复动量状态 |
| **仅推理/部署** | ❌ 不需要 | 只需模型权重 |
| **微调新任务** | ❌ 不需要 | 创建新优化器 |

**原因**：Adam 优化器维护**内部状态**：
```python
# Adam 的内部状态（每个参数都有）
m_t: 一阶动量（梯度的移动平均）
v_t: 二阶动量（梯度平方的移动平均）
step: 步数计数器
```

**类比开车**：
- **模型参数** = 当前位置
- **优化器状态** = 车速 + 方向 + 加速度历史

如果不保存优化器状态：
- 重启训练时"车速归零"
- 需要重新加速，训练曲线会抖动
- 浪费几百到几千步重新积累动量

**最佳实践**：
```python
# 训练中定期保存（完整 checkpoint）
torch.save({
    "model_state_dict": model.state_dict(),
    "optimizer_state_dict": optimizer.state_dict(),  # 保存
}, "checkpoint.pth")

# 最终部署（只保存模型）
torch.save(model.state_dict(), "model_final.pth")  # 不保存优化器
```

## 17. 为什么保存优化器后文件从 653MB 变成 1.95GB？

**答案**：Adam 优化器需要存储 **2 份动量**，每份大小和模型参数相同。

**文件大小组成**：
```
总大小 = 模型参数 + 优化器状态
1.95GB ≈ 0.65GB  + 1.3GB
```

**详细拆解**（以 163M 参数为例）：
```python
模型参数 θ:     163M × 4 bytes = 652 MB
一阶动量 m_t:   163M × 4 bytes = 652 MB  ← Adam 状态
二阶动量 v_t:   163M × 4 bytes = 652 MB  ← Adam 状态
--------------------------------------------
总计:           163M × 12 bytes ≈ 1.96 GB (3倍)
```

**Adam 为什么需要两份动量？**
- **m_t（一阶动量）**：记录梯度的"方向和速度"
- **v_t（二阶动量）**：记录梯度的"波动程度"（用于自适应学习率）

**Load 到内存的大小**：
```python
# 场景1：只推理
model.load_state_dict(checkpoint["model_state_dict"])
# 内存: 652 MB

# 场景2：继续训练
model.load_state_dict(checkpoint["model_state_dict"])
optimizer.load_state_dict(checkpoint["optimizer_state_dict"])
# 内存: 1.95 GB

# 场景3：训练时（更大！）
# 内存: 1.95 GB + 梯度(652 MB) + 激活值(1-3 GB) ≈ 4-6 GB
```

**优化存储的方法**：
1. 使用 SGD + Momentum（2倍而非3倍）
2. 使用 8-bit Adam（节省50%）
3. 只保留最近 N 个 checkpoint

## 18. GPT-2 124M 参数为什么文件是 652.9MB 而不是 496MB？

**验证参数量**：
```python
标准 GPT-2 Small 配置:
emb_dim = 768
n_heads = 12
n_layers = 12
vocab_size = 50257

# 标准计算
token_emb = 50257 × 768 = 38.6M
pos_emb = 1024 × 768 = 0.79M
transformer = 12 层 × 7.08M/层 = 84.97M
总计 ≈ 124.4M 参数

# 存储大小（float32）
124.4M × 4 bytes = 497.6 MB ≈ 500 MB
```

**652.9MB 反推参数量**：
```python
652.9 MB ÷ 4 bytes = 163M 参数

# 这不是标准的 124M！
```

**可能原因**：
1. **配置不同**：
   ```python
   # 可能是扩展配置

   ```

2. **包含优化器状态**：如果是完整 checkpoint
3. **不同模型版本**：介于 Small 和 Medium 之间

**实际文件大小观察**：
```bash
model_and_optimizer.pth: 1.8GB
# 这确实是 模型(~600MB) + 优化器(~1.2GB)
```

**关键区别**：
- **参数量（个数）**：124M = 1.24 亿个浮点数

## 19. Python 数字中的下划线（151_936）是什么意思？

**答案**：这是 **Python 3.6+ 的数字视觉分隔符**，用于提高大数字可读性。

**效果**：
```python
# 以下完全等价
vocab_size = 151936      # 难以快速识别
vocab_size = 151_936     # 一眼看出：15万1936
vocab_size == 151936     # True
```

**作用**：
- 纯视觉分隔，不影响数值
- 类比：日常写数字时的逗号（1,000,000）

**使用规则**：
```python
# ✅ 推荐：每三位一组（千位分隔）
population = 7_800_000_000  # 78亿

# ✅ 二进制/十六进制按4位分组
flags = 0b_1111_0000_1010_0101
color = 0xFF_AA_00

# ✅ 浮点数
pi = 3.141_592_653_589

# ❌ 不能在开头或结尾
invalid = _1000  # SyntaxError
invalid = 1000_  # SyntaxError
```

**为什么要用？**
```python
# 对比效果
before = 151936        # 需要数：1..5..1..9..3..6（6位数？）
after  = 151_936       # 直接看出：15万+1936

# 在配置中
"vocab_size": 151_936,      # 清晰：15万级别
"context_length": 40_960,   # 清晰：约4万
"rope_base": 1_000_000.0,   # 清晰：100万
```

**其他语言的支持**：
- Java (7+), C++ (14+), Rust, JavaScript (ES2021) 都支持类似语法

## 20. emb_dim 和 hidden_dim 的区别？hidden_dim 是 MLP 的最高维度吗？

**完全正确！**

| 参数 | 含义 | 在 Qwen 0.6B 中 |
|------|------|-----------------|
| **emb_dim** | 模型主干维度（每个 token 的向量维度） | 1024 |
| **hidden_dim** | FFN 中间扩展维度（MLP 的最高维度） | 3072 (3倍) |

**数据流**：
```python
输入: [batch, seq_len, emb_dim=1024]
  ↓
Multi-Head Attention
  维度保持: 1024 → 1024
  ↓
Feed-Forward Network (FFN)
  ┌─────────────────────┐
  │ Linear1: 1024 → 3072 │  ← hidden_dim（扩展）
  │ Activation (GELU)     │
  │ Linear2: 3072 → 1024 │  ← 压缩回原维度
  └─────────────────────┘
  ↓
输出: [batch, seq_len, emb_dim=1024]
```

**为什么要"扩展-压缩"？**
- **扩展**：在高维空间中更容易学习复杂的非线性模式
- **激活**：非线性变换提取特征
- **压缩**：蒸馏信息，保留最重要的特征

**参数量占比**：
```python
单层参数量:
- Attention: ~4.2M
- FFN:       ~6.3M  ← 占 60%！

FFN 参数 = 2 × emb_dim × hidden_dim
         = 2 × 1024 × 3072
         = 6,291,456
```

**扩展倍数的设计**：
| 模型 | emb_dim | hidden_dim | 倍数 |
|------|---------|-----------|------|
| GPT-2 | 768 | 3072 | 4x |
| Qwen 0.6B | 1024 | 3072 | 3x |
| LLaMA 7B | 4096 | 11008 | ~2.7x |

**类比**：
- **emb_dim**：高速公路（固定宽度，快速通行）
- **hidden_dim**：服务区广场（临时扩展，复杂操作）

## 21. head_dim 是每一个 attention 头的维度吗？

**完全正确！**

**标准关系**：
$$\text{emb\_dim} = \text{n\_heads} \times \text{head\_dim}$$

**Qwen 0.6B 配置验证**：
```python
emb_dim = 1024
n_heads = 16
head_dim = 128

# 验证
1024 == 16 × 128  ✓
```

**维度流动**：
```python
输入: [batch, seq_len, emb_dim=1024]
  ↓
Query 投影: 1024 → (n_heads × head_dim) = 2048
  ↓
重塑为多头: [batch, seq_len, 16, 128]
             转置→ [batch, 16, seq_len, 128]
                           ↑           ↑
                           |           └─ 每个头处理 128 维
                           └─ 16 个独立的头
  ↓
每个头独立计算注意力（在 128 维空间中）
  ↓
合并: [batch, 16, seq_len, 128] → [batch, seq_len, 2048]
  ↓
输出投影: 2048 → 1024
```

**Qwen 的特殊设计（GQA）**：
```python
# 标准 MHA
head_dim = emb_dim / n_heads = 1024 / 16 = 64

# Qwen GQA
head_dim = 128  # 显式指定（2倍！）

# Query 维度扩展
Q_dim = n_heads × head_dim = 16 × 128 = 2048 (> emb_dim)

# Key/Value 共享（节省 KV 缓存）
n_kv_groups = 8  # 只有 8 组 KV
KV_dim = n_kv_groups × head_dim = 8 × 128 = 1024
```

**为什么所有 Qwen 模型都用 head_dim=128？**
1. **统一设计**：跨模型规模复用 RoPE 参数
2. **硬件对齐**：128 是 2 的幂次，GPU 友好
3. **更大表征空间**：每个头能表示更丰富的信息
4. **与 GQA 配合**：通过 KV 共享平衡内存和性能

**head_dim 必须是偶数**：
- RoPE（旋转位置编码）需要成对旋转维度
- 如 128 维 → 64 对 → 每对做 2D 旋转

**类比**：
- **n_heads（16）**= 16 个侦察兵
- **head_dim（128）**= 每个侦察兵的记事本有 128 页
- 侦察兵各自观察（独立注意力），最后汇报合并
